Summary

This sample is an XWorm RAT delivered under the filename e-dekont_html.exe, "dekont" being Turkish for payment receipt, suggesting a fake-invoice lure targeting Turkish-speaking users. It's a .NET-based remote access trojan, active in the wild since at least 2022, offering keylogging, remote desktop, credential theft, and full backdoor command execution.

I pulled the sample from MalwareBazaar and detonated it in a cloud sandbox (Recorded Future Triage) to observe its actual runtime behavior rather than relying on the vendor description alone.

Execution Chain

The process tree is where this sample gets interesting. Instead of just running its payload directly, it uses Windows' own, legitimately signed .NET compiler tools to build a second copy of itself on the victim machine at runtime:

  1. e5dda4e7dc6b...e7.exe (the original dropped file) launches jsc.exe, a legitimate Microsoft JScript compiler component.
  2. jsc.exe launches csc.exe, the actual C# compiler, and hands it a dropped source file: guardian_src_febe1644...cs.
  3. csc.exe compiles that source into a new executable, written to %TEMP%\jsc.exe (reusing the filename, not the original binary).
  4. That freshly compiled file is then executed. This is the real second-stage payload, built locally instead of dropped pre-compiled.

Since csc.exe and jsc.exe are legitimate, signed Microsoft binaries, this technique (sometimes called a LOLBin, Living off the Land Binary) is far less likely to trip signature or allowlisting-based defenses than dropping and running an unsigned executable directly.

Persistence, Two Layers

The sample doesn't rely on a single persistence mechanism. It stacks two:

1. File relocation. The malware copies itself out of the original %TEMP% location into C:\ProgramData\AppData\, a folder that survives routine temp-file cleanup and blends in better with legitimate installed software.

2. Registry Run key. It then adds a Run key pointed specifically at that ProgramData copy, not the original file:

HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\AppData =
  "cmd /c start "" /D "C:\ProgramData\AppData" e5dda4e7dc6b...e7.exe"

The result is redundant, durable persistence. Even if the original Temp file gets cleaned up, the relocated copy and its Run key survive independently.

Other Observed Behavior

  • Outlook profile access: the sample opens registry keys tied to Outlook mail profiles, consistent with XWorm's known email/credential collection capability.
  • Anti-sandbox checks: queries SCSI disk registry keys and processor information, both common techniques malware uses to detect if it's running inside a VM or sandbox.
  • Geofencing: checks the system's configured country code, suggesting the operator may restrict or vary behavior by victim region.
  • SetThreadContext / WriteProcessMemory: flagged as suspicious by the sandbox, consistent with process injection techniques XWorm is known to use.

Network

The sample connects out to 203.202.232.132:2828 over raw TCP, not HTTP. XWorm uses a custom binary protocol for C2 communication, AES-encrypted configuration and command traffic with a periodic ping/pong keepalive, rather than a web-based panel. A basic TCP connectivity check confirmed the port was open and actively accepting connections at time of analysis.
